# 2026-07-16: sequencing toward the adoption demo

Status: agreed direction. Companion to [`design.md`](design.md); this document decides what to do
next, given that a demo is wanted soon and the host/adoption flow must be part of it.

## The call

No new slices, and no broad polish across all existing slices. Go **depth on one slice plus the
adoption seam**: for this demo the adoption flow is the headline feature, and Data Authoring is
the one slice that has to carry it.

- **New slices: no.** Every domain added before the seam exists increases the composition
  knowledge an adopter must reverse-engineer, and dilutes demo quality across more surface. New
  slices multiply in value _after_ the manifest/adoption machinery exists, because each one lands
  as an adoption target from day one.
- **Finish-everything-first: no.** One excellent vertical beats four decent ones, and the showcase
  home already sells four stories. Data Authoring is the flagship, the furthest along, and the
  adoption proving case named in `design.md`. Game Text and Texture Audit are demo-adequate as
  supporting cast; Map Review and Camera Lab get no investment before this demo.
- **Plan 007 waits, again for the migrate-twice reason.** The repository already paused 007 once
  so conflicts/rich types would not be built on pre-Effect services. The same logic now applies to
  the host seam: 007 is XL and would grow workflow code inside the current Workbench-service
  shape, which plan 016 would then relocate. 007 lands on the new shape, after the demo. The
  current draft → apply → save loop already demos honestly without it.

## Order of work

1. **Finish the Effect track (plans 014, 015).** Non-negotiable foundation; 014 is mid-flight in
   the working tree.
2. **Finish plans 004 and 005 (both IN PROGRESS).** The session service and the Peculiar Sheets
   draft editor are not optional polish — they are the flagship slice's UX. If they are rough, the
   demo is rough regardless of the adoption story.
3. **Plan 016, demo-scoped.** The demo-critical path is narrower than the full `design.md` scope:
    - a direct in-process `AuthoringClientLive`;
    - a slim `ShedHostLive` covering the authoring path;
    - the minimal non-Electron example host;
    - the Data Authoring slice manifest plus per-slice agent doc.

    Trailing after the demo (same seam, more of it): the generalized transport bridge, the CLI
    re-adapter rewrite, manifests for the remaining slices, the camera-lab slice move, and the
    complete Workbench refactor.

4. **Run the fresh-agent adoption eval before the demo, not during it.** Live agent runs are
   nondeterministic and slow — the wrong thing to bet a room on. Pre-run it and demo the artifact:
   the blank project, the manifest, the slice running inside it. For live proof of ownership,
   edit the copied StyleX tokens in the adopted project and watch it restyle — divergence made
   visible in thirty seconds, no agent latency involved.
5. **After the demo:** plan 007 on the new shape, then new slices as adoption targets.

## Demo script this buys

The existing showcase, unchanged — then "here is what kind of library this actually is": open a
project that is _not_ this repository, with the authoring slice running in it, restyled, alongside
the manifest and conformance run that got it there. The second act is the differentiator, and it
only needs one slice to be true.

## Risk to retire first

The least-proven assumption in the whole design is that the StyleX/Solid toolchain wiring in a
foreign Vite project is smooth. Spike exactly that — adopt the authoring slice into a scratch
project by hand, once — **before committing the demo date**. If it is gnarly, the fallback
(precompiled CSS for the demo build) needs lead time.
